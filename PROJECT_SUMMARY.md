## 📝 面試準備問題

### Q1: 為什麼選擇 Firestore 而不是傳統 SQL？

**回答**：
> 「Firestore 是 NoSQL 資料庫，適合儲存非結構化的測試結果資料。
> 而且 GCP 免費層提供了足夠的額度（每天 50K 讀取），對於個人專案完全夠用。
> 如果未來需要更複雜的查詢，可以考慮遷移到 Cloud SQL。」

### Q2: 如何確保測試結果的可靠性？

**回答**：
> 「我實作了多層驗證：
> 1. Playwright 的內建等待機制確保元素載入完成
> 2. 測試結果自動上傳到 Firestore，避免本地檔案遺失
> 3. GitHub Actions 提供完整的執行日誌和錯誤追蹤
> 4. API 提供查詢介面，方便後續分析和除錯。」

### Q3: 如果測試量增加，如何擴展？

**回答**：
> 「目前的架構已經考慮了擴展性：
> 1. Firestore 可以輕鬆處理更大的資料量（升級到付費層）
> 2. GitHub Actions 支援平行執行多個測試任務
> 3. 可以將測試分散到多個 Cloud Run 實例執行
> 4. API 使用 FastAPI，支援非同步處理，可以處理高併發請求。」
